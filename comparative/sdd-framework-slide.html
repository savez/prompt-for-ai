01 Identità e Filosofia

↓ dettagli per ogni framework

Spec Kit github/spec-kit · Python/uv · ⭐ 81.4k

Spec-Driven Development

CLI specify con 7 fasi sequenziali: constitution → specify (COSA, no tech) → clarify (domande) → plan (tech stack) → analyze → tasks → implement. Ogni fase produce Markdown versionato in .specify/.

Differenziatore: La fase clarify forza domande strutturate prima del piano. Extensions + presets stackabili (Agile, DDD, compliance). 25+ agent.

Superpowers obra/superpowers · Shell/JS · ⭐ 137k

Agentic Skills Framework

Skill auto-triggered: brainstorming socratico → worktree git → subagent per task con two-stage review → TDD imposto (se scrivi codice prima dei test, le istruzioni dicono di cancellarlo). Zero comandi.

Differenziatore: Subagent-driven, autonomia per ore. Debugging strutturato 4 fasi. Le skill sono prompt nel contesto, non un runtime — attivazione non garantita tecnicamente.

OpenSpec Fission-AI/OpenSpec · TypeScript/npm · ⭐ 37.7k

"Fluid, not rigid. Built for brownfield."

/opsx:propose "feature" genera in un colpo: proposal.md, specs/, design.md, tasks.md. Nessun phase gate — modifica qualsiasi artefatto in qualsiasi momento. Archivia con /opsx:archive.

Differenziatore: Velocità massima. Idea → piano in un comando. Brownfield-first: ogni change isolata.

B-MAD bmad-code-org/BMAD-METHOD · Node.js 20+ · ⭐ 42.1k

Agentic Agile Driven Development

19+ agenti specializzati (Analyst, PM, Architect, Dev, QA) ognuno con persona e comandi propri. Scale-adaptive: Quick (bug fix) · BMad (PRD+Arch+UX) · Enterprise (+Security/DevOps/Test).

Differenziatore: Multi-agent reale. Adversarial review. Unico non limitato al software (game dev, creative, business). Curva apprendimento più ripida.

03 Punti di Forza

Spec Kit

  • Rigore — 7 step con gate
  • Clarify — rimuove ambiguità prima del plan
  • Extensions/Presets stackabili a 3 livelli
  • Walkthroughs — ASP.NET 307k LOC, Java 420k LOC

Superpowers

  • Autonomia — ore senza deviare
  • TDD via istruzioni — RED→GREEN→REFACTOR
  • Subagent + two-stage review
  • Debugging strutturato 4 fasi

OpenSpec

  • Fluidità — modifica tutto, sempre
  • Brownfield-first
  • Setup istantaneo — npm + init
  • Archiviazione automatica ordinata

B-MAD

  • Multi-agent reale — 19+ con persona
  • Scale-adaptive — Quick / BMad / Enterprise
  • Adversarial review cross-agente
  • Ecosystem — Game Dev, CIS, Builder

04 Workflow

↓ un workflow per slide

Spec Kit

1specify init . --ai claude
2/speckit.constitution — principi fondamentali
3/speckit.specify — COSA (no tech stack)
4/speckit.clarify — rimuove ambiguità con domande
5/speckit.plan — COME (tech stack, architettura)
6/speckit.analyze — consistency check
7/speckit.tasks → /speckit.implement

Gate rigidi. Separazione "cosa" vs "come". Ogni fase → Markdown versionato.

Superpowers

1Plugin install
2Parli dell'idea AUTO:brainstorming
3Approvi design AUTO:worktree
4"Go" AUTO:subagent → two-stage review
5Codice AUTO:TDD RED→GREEN→REFACTOR
6Fine AUTO:finish-branch

Zero comandi. Skill = prompt nel contesto, non runtime.

OpenSpec

1openspec init
2/opsx:propose "feature" → 4 artefatti
3Modifica qualsiasi artefatto liberamente
4/opsx:apply → /opsx:verify → /opsx:archive

Un comando genera tutto. Nessun phase gate. Brownfield-first.

B-MAD

1npx bmad-method install
2Analyst PERSONA brainstorm + elicitation
3PM → PRD + stories
4Architect → design tecnico
5Dev → implementa · QA → adversarial review

Team agile simulato. 3 track: Quick / BMad / Enterprise.

06 Coesistenza

Premessa: Superpowers = prompt nel contesto, non runtime. Comandi espliciti (slash) vincono sulle skill implicite.
⚠ TEORICA

Spec Kit + Superpowers — Nessun conflitto file. Skill non garantite durante slash commands.

⚠ BASSA

Spec Kit + OpenSpec — Due set di artefatti paralleli. L'agente non sa quale seguire.

⚠ SOVRAPPOS.

Spec Kit + B-MAD — Due metodologie complete in parallelo = confusione.

⚠ TEORICA

Superpowers + B-MAD — Conflitto su "chi comanda" nel contesto.

✅ POSSIBILE

OpenSpec + B-MAD — In alternanza: OpenSpec per change leggere, B-MAD per complesse.

❌ NO

Tutti insieme — Troppe istruzioni contrastanti.

07 Casi d'Uso

↓ uno scenario per slide · Quale tool e perché non gli altri

Greenfield, requisiti vaghi

→ Spec Kit

Situazione: "Serve un servizio prenotazioni". Non sai quante API, quale DB, come le notifiche.

Perché: specify → clarify (domande fino a risolvere tutte le ambiguità) → plan (ORA tech stack). Ti forza a decidere cose che altrimenti scopriresti con un bug.

Non gli altri: Superpowers no artefatti formali. OpenSpec non forza a chiarire. B-MAD curva apprendimento troppo ripida per partire.

Feature su codebase esistente

→ OpenSpec

Situazione: Aggiungere dark mode, endpoint, integrazione. Conosci il codice.

Perché: /opsx:propose "add-dark-mode" → 30 sec, 4 artefatti. Scopri vincoli? Aggiorni il design senza ricominciare.

Non gli altri: Spec Kit 7 step per dark mode = overkill. Superpowers no artefatti per colleghi. B-MAD overhead per task noti.

Focus time — Claude Code lavora da solo

→ Superpowers

Situazione: Hai chiaro cosa costruire. Vuoi che l'agente lavori in autonomia per ore.

Perché: Subagent per task, review 2 stadi, TDD imposto. Puoi andare a fare altro.

⚠ Solo Claude Code — su altri agent non garantito.

Non gli altri: Spec Kit e OpenSpec no subagent/review. B-MAD checkpoint umano tra fasi.

Progetto complesso, molti stakeholder

→ B-MAD (BMad / Enterprise track)

Situazione: FE+BE+API+DB, requisiti security, più persone devono capire le decisioni architetturali.

Perché: Analyst elicita → PM genera PRD → Architect progetta → Dev implementa → QA adversarial review. Ogni ruolo = un agente. Scale-adaptive.

Non gli altri: Spec Kit un agente fa tutto. Superpowers no PRD/arch doc. OpenSpec troppo leggero.

Bug fix / task piccolo

→ B-MAD Quick o nessuno

Unico con track per task piccoli: solo tech-spec. Per un typo → nessun framework.

Spec Kit 7 step? No. OpenSpec 4 file per 5 righe? No.

Bug complesso, root cause ignota

→ Superpowers

Unico con debugging strutturato: raccolta evidenze → ipotesi → verifica sistematica → fix defense-in-depth.

Gli altri non coprono debugging come workflow.

Compliance / audit

→ Spec Kit + presets

Constitution → Spec → Plan → Tasks in Git. Presets per gate regolatori. B-MAD Enterprise alternativa.

Side project

→ OpenSpec

npm + init + 3 comandi. Quando cresce, hai già artefatti.

Non-software (game dev, creative, strategy)

→ B-MAD (unica opzione)

Game Dev Studio (4 agenti, 20+ workflow), Creative Intelligence Suite, business strategy. Gli altri tre sono solo software development.
