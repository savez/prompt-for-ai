1 Identità e Filosofia

Spec Kit

Python/uvGitHub Official

Spec-Driven Development — le specifiche diventano eseguibili. Fasi rigide: constitution → specify → plan → tasks → implement. Artefatti Markdown in .specify/.

Superpowers

Shell/JSJesse Vincent

Agentic Skills Framework — skill composabili auto-triggered dall'agente. Zero ceremony. Enfasi fortissima su TDD e YAGNI. L'agente rileva il contesto e agisce.

OpenSpec

TypeScript/npmFission AI

Fluid, not rigid. Spec-driven ma anti-cerimoniale. Ogni change ha la sua cartella. Nessun phase gate. Brownfield-first.

B-MAD

Node.js/npmBMad Code

Agentic Agile Driven Development — simula un team agile completo con agenti specializzati (Analyst, PM, Architect, Dev, QA). Scale-adaptive: si adatta da bug fix a enterprise. 19+ agenti, 50+ workflow, moduli espandibili.

3 In Cosa Eccellono

Spec Kit

  • Rigore del processo — 7 step con gate espliciti
  • Clarification workflow — /speckit.clarify rimuove ambiguità prima del plan
  • Extensions/Presets — overrides > presets > extensions > core
  • Walkthroughs enterprise — ASP.NET 307k LOC, Java 420k LOC
  • 25+ agent supportati nativamente

Superpowers

  • Autonomia massima — lavora ore senza deviare
  • TDD hardcoded — cancella codice scritto prima dei test
  • Subagent-driven — two-stage review (spec + quality)
  • Debugging strutturato — 4 fasi, root cause tracing
  • Zero ceremony — nessun comando da ricordare

OpenSpec

  • Fluidità massima — modifica qualsiasi artefatto in qualsiasi momento
  • Brownfield-first — ogni change isolata in cartella
  • Setup istantaneo — npm + init
  • Dashboard visuale — stato delle change
  • Archiviazione — storia ordinata automaticamente

B-MAD

  • Multi-agent reale — ogni agente ha persona, principi, e comandi propri
  • Scale-adaptive — 3 track (Quick/BMad/Enterprise) in base alla complessità
  • Adversarial review — verifica cross-agente deliberatamente critica
  • Ecosystem modulare — Game Dev, Creative Intelligence, Builder per creare i tuoi moduli
  • Non solo software — business strategy, entertainment, wellness

4 Workflow Step-by-Step

Spec Kit

1specify init . --ai claude
2/speckit.constitution — principi di progetto
3/speckit.specify — COSA vuoi (no tech stack)
4/speckit.clarify — rimuove ambiguità
5/speckit.plan — COME (tech stack, architettura)
6/speckit.analyze — consistency check
7/speckit.tasks → /speckit.implement
Separazione netta "cosa" vs "come". Phase gate rigidi. Ogni fase produce artefatti versionati.

Superpowers

1Plugin install
2Parli dell'idea AUTO: brainstorming
3Approvi design AUTO: git-worktrees
4Piano auto AUTO: writing-plans
5Dici "go" AUTO: subagent-dev
6Codice AUTO: TDD RED→GREEN→REFACTOR
7Fine AUTO: finish-branch
Zero comandi. L'agente decide le transizioni. Skill sono prompt nel contesto, non runtime.

OpenSpec

1openspec init
2/opsx:propose "feature" — genera 4 artefatti
3Modifica qualsiasi artefatto liberamente
4/opsx:apply → /opsx:verify
5/opsx:archive
Un comando genera tutto. Nessun phase gate. Modifica spec durante l'implementazione.

B-MAD

1npx bmad-method install
2Carica Analyst Agent PERSONA — brainstorm + elicitation avanzata
3PM Agent genera PRD + user stories
4Architect Agent → architettura + tech design
5Dev Agent → implementazione story-centric
6QA Agent → test + adversarial review
7bmad-help skill guida in qualsiasi momento
Simula un team agile. Ogni agente ha persona, principi, comandi. 3 track di complessità: Quick (tech-spec only), BMad (PRD+Arch+UX), Enterprise (+ Security/DevOps/Test).

6 Confronto del Planning

Processo Formale

Controllo: TU. L'agente genera, tu validi.

Forza: /speckit.clarify forza domande PRIMA del plan.

Limite: Cerimonia completa pesante per feature piccole.

Conversazione Naturale

Controllo: L'agente propone, tu validi. Decide quando passare avanti.

Forza: Two-stage review garantisce aderenza al piano.

Limite: Tracciabilità informale. Non basta per audit.

Proposta Fluida

Controllo: TU, con meno cerimonia. /opsx:propose genera tutto.

Forza: Velocità. Idea → piano in un comando.

Limite: Senza gate, rischio di spec incomplete.

Team Agile Simulato

Controllo: Ogni agente ha un ruolo. L'Analyst elicita, il PM formalizza, l'Architect progetta.

Forza: Scale-adaptive: Quick track salta la cerimonia, Enterprise track la espande.

Limite: Curva di apprendimento. 19+ agenti e 50+ workflow da capire. Overhead per task semplici se non usi Quick track.

7 Possono Coesistere?

Come funziona l'"auto-trigger" di Superpowers: Non c'è un runtime o daemon. Sono prompt iniettati nel contesto dell'agente. L'attivazione non è osservabile — verificala dal comportamento (fa brainstorming? impone TDD?).
Problema della priorità: Con più tool installati, i comandi espliciti (slash) tendono a vincere sull'attenzione dell'agente rispetto alle istruzioni implicite (skill). Il comportamento non è deterministico.
⚠️ TEORICA

Spec Kit + Superpowers

Nessun conflitto file system. Ma le skill di Superpowers non hanno garanzia di attivarsi durante i comandi di Spec Kit. Possibile conflitto brainstorming/clarify e writing-plans/speckit.plan.

⚠️ BASSA

Spec Kit + OpenSpec

Competono sullo stesso terreno: due set di artefatti paralleli per la stessa feature. L'agente non sa quale seguire.

⚠️ TEORICA

Superpowers + OpenSpec

Stessa dinamica di Spec Kit + Superpowers. Nessun conflitto directory. Stessa incertezza sull'attivazione skill durante /opsx:apply.

⚠️ SOVRAPPOSIZIONE

Spec Kit + B-MAD

Entrambi producono spec formali (spec.md vs PRD) e piani strutturati. B-MAD ha agenti con ruoli definiti che sovrappongono le fasi di Spec Kit. Due "metodologie complete" nello stesso progetto = confusione.

⚠️ TEORICA

Superpowers + B-MAD

B-MAD ha i propri agenti con persona e principi; Superpowers ha skill auto-triggered. Entrambi vogliono guidare il comportamento dell'agente in modo diverso. Conflitto di "chi comanda" nel contesto. B-MAD ha già il suo QA agent vs il TDD di Superpowers.

✅ POSSIBILE

OpenSpec + B-MAD

L'unica combinazione con potenziale reale: OpenSpec per change leggere e veloci (bug fix, small feature) dove B-MAD sarebbe overkill. B-MAD per feature complesse dove serve il team agile simulato. Usati in alternativa sullo stesso progetto, non contemporaneamente sulla stessa feature.

❌ SCONSIGLIATO

Tutti insieme

Troppe istruzioni contrastanti. Quattro metodologie che competono per l'attenzione dell'agente.

8 Casi d'Uso: Quale Scegliere e Perché

🆕 Nuovo progetto greenfield con architettura complessa

→ Spec Kit oppure → B-MAD (Enterprise track)

Perché Spec Kit: La separazione netta "cosa" (specify) vs "come" (plan) ti forza a chiarire i requisiti senza farti influenzare dalla tecnologia. /speckit.clarify elimina ambiguità prima di scrivere una riga. La constitution governa tutte le decisioni. Ideale se vuoi controllo totale e audit trail.

Perché B-MAD: Se il progetto è davvero complesso (multi-repo, team distribuito, requisiti security), l'Enterprise track aggiunge Security/DevOps/Test strategy. Gli agenti specializzati (Analyst, Architect) producono artefatti più ricchi di un singolo agente guidato da slash commands. L'adversarial review cattura problemi architetturali.

Quando NON questi: Se è un prototipo rapido, entrambi sono overkill. Usa OpenSpec o nessun framework.

🔧 Feature su codebase esistente (brownfield)

→ OpenSpec oppure OpenSpec + B-MAD (Quick track) in alternanza

Perché OpenSpec: Esplicitamente brownfield-first. La struttura per-change isola ogni modifica. La fluidità permette di scoprire vincoli durante l'implementazione e aggiornare la spec senza ricominciare. Per change piccole e medie è la scelta più efficiente.

Quando aggiungere B-MAD: Se la feature brownfield è complessa (cambia architettura, tocca molti moduli), il Quick track di B-MAD produce un tech-spec veloce senza tutta la cerimonia. Usali in alternanza: OpenSpec per change normali, B-MAD Quick per quelle complesse.

⚡ Sessione di sviluppo intensiva con Claude Code

→ Superpowers

Perché: Il subagent-driven development permette a Claude di lavorare per ore in autonomia. Le skill auto-triggered eliminano il context-switching. Il TDD hardcoded garantisce codice testato. Nessun altro framework offre questo livello di autonomia dell'agente.

Limite: Funziona al meglio solo su Claude Code (plugin marketplace nativo). Su altri agent l'attivazione delle skill non è garantita.

🏢 Progetto enterprise con compliance / audit

→ Spec Kit oppure → B-MAD (Enterprise track)

Perché Spec Kit: Constitution + presets compliance + checklist di qualità. Ogni decisione tracciata in Markdown versionato. Il sistema di presets permette di aggiungere gate regolatori (V-Model, traceability). 118 release, backed da GitHub.

Perché B-MAD: L'Enterprise track aggiunge nativamente Security e DevOps strategy. Gli agenti specializzati producono artefatti che mappano ruoli enterprise reali (BA → PM → Architect → Dev → QA). L'adversarial review è un differenziatore forte per contesti regolati.

🐛 Bug fix o task molto piccolo

→ B-MAD (Quick track) oppure nessun framework

Perché B-MAD Quick: È l'unico con un track esplicito per task piccoli. Genera solo un tech-spec, niente PRD/architettura/UX. Minima cerimonia utile.

Quando nessuno: Per un bug fix di 5 minuti, qualsiasi framework è overhead. Fai il fix, scrivi il test, committa.

🎮 Progetto non-software (game dev, creative, strategy)

→ B-MAD (unica opzione)

Perché: È l'unico dei quattro pensato per domini oltre il software. Il modulo Game Dev Studio ha 4 agenti dedicati e 20+ workflow per game development. Il Creative Intelligence Suite copre brainstorming creativo. Gli altri tre sono esclusivamente per software development.

🧪 Prototipo rapido / spike

→ OpenSpec oppure nessun framework

Perché OpenSpec: Un singolo /opsx:propose ti dà struttura minima senza cerimonia. Se lo spike diventa feature, hai già gli artefatti. /opsx:archive archivia gli spike scartati.

Quando nessuno: Se lo spike è "vediamo se funziona in 30 minuti", nessun framework serve.

🔍 Debugging complesso in produzione

→ Superpowers

Perché: Unico con skill di debugging strutturato: 4 fasi con root cause tracing, defense-in-depth, condition-based waiting. Gli altri non coprono esplicitamente il debugging come workflow.
