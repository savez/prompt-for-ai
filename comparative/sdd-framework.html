1 Identità e Filosofia

Spec Kit

Python / uv GitHub Official MIT

Spec-Driven Development — le specifiche diventano eseguibili. Il codice segue la spec, non il contrario.

Strutturato con fasi rigide: constitution → specify → plan → tasks → implement. Ogni fase produce artefatti Markdown versionati in .specify/.

Superpowers

Shell / JS Jesse Vincent MIT

Agentic Skills Framework — skill composabili che l'agente attiva automaticamente. Non serve invocare comandi: l'agente "ha i superpoteri".

Behavior-driven. L'agente rileva il contesto e attiva la skill appropriata. Enfasi fortissima su TDD (RED-GREEN-REFACTOR) e YAGNI.

OpenSpec

TypeScript / npm Fission AI MIT

Fluid, not rigid. Iterative, not waterfall. Spec-driven ma anti-cerimoniale. Aggiorna qualsiasi artefatto in qualsiasi momento.

Artifact-guided: ogni change ha la sua cartella con proposal, specs, design e tasks. Leggero e orientato al brownfield.

3 In Cosa Eccellono

Spec Kit

  • Rigore del processo — ogni fase è un gate esplicito. Il team sa sempre dove si trova.
  • Documentation enterprise — constitution.md come "legge fondamentale". Ottimo per compliance.
  • Clarification workflow — /speckit.clarify forza la rimozione di ambiguità PRIMA del plan. Riduce il rework.
  • Extensions/Presets — sistema stratificato: overrides > presets > extensions > core. Supporta Agile, Kanban, DDD, regulatory.
  • Walkthroughs reali — brownfield ASP.NET 307k LOC, Java 420k LOC / 180 moduli Maven.

Superpowers

  • Autonomia dell'agente — Claude Code lavora per ore senza deviare. Le skill sono mandatory, non suggerimenti.
  • TDD nativo — RED-GREEN-REFACTOR hardcoded. Cancella codice scritto prima dei test.
  • Subagent-driven development — subagent fresh per task con two-stage review (spec + quality).
  • Debugging strutturato — 4 fasi, root cause tracing, defense-in-depth, condition-based waiting.
  • Zero ceremony — nessun comando da ricordare. L'agente rileva il contesto e agisce.

OpenSpec

  • Fluidità — nessun phase gate. Modifica la proposal mentre stai già implementando.
  • Brownfield-first — progettato per aggiungere feature a codebase esistenti.
  • Setup veloce — npm install -g + openspec init. Operativo in secondi.
  • Archiviazione ordinata — /opsx:archive mantiene la storia pulita.
  • Dashboard visuale — overview dello stato di tutte le change.

4 Workflow Step-by-Step

Spec Kit

1specify init . --ai claude — scaffolda .specify/
2/speckit.constitution — principi di progetto (constitution.md)
3/speckit.specify — COSA vuoi (user stories, no tech stack)
4/speckit.clarify — domande strutturate per rimuovere ambiguità
5/speckit.plan — COME farlo (tech stack, architettura)
6/speckit.analyze — cross-artifact consistency check
7/speckit.tasks — task breakdown con dipendenze e [P]
8/speckit.implement — esegue tutto in ordine
Separazione netta tra "cosa" (specify) e "come" (plan). Il plan è dove dichiari il tech stack. Prima del plan, la conversazione è tech-agnostic.

Superpowers

1Installa plugin SETUP
2Parli di cosa costruire AUTO: brainstorming — domande socratiche, alternative, design in chunk
3Approvi il design AUTO: git-worktrees — branch isolato, setup, verifica test baseline
4Piano auto-generato AUTO: writing-plans — task da 2-5 min con file path e codice completo
5Dici "go" AUTO: subagent-dev — subagent per task → review stage 1 (spec) → review stage 2 (quality)
6Implementazione AUTO: TDD — RED → GREEN → REFACTOR (cancella codice pre-test)
7Tra i task AUTO: code-review — issue per severità, critiche bloccano
8Fine AUTO: finish-branch — merge/PR/keep/discard + cleanup
Nessuna separazione rigida. Il brainstorming scorre nel design, che scorre nel planning. L'agente decide le transizioni. Il piano è scritto per "un junior entusiasta con scarso gusto" — istruzioni ultra-dettagliate.

OpenSpec

1openspec init — genera config per l'agent
2/opsx:propose "feature" — genera in un colpo: proposal.md, specs/, design.md, tasks.md
3Modifica qualsiasi artefatto in qualsiasi momento (nessun phase gate)
4/opsx:apply — implementa i task dalla checklist
5/opsx:verify — verifica spec ↔ implementazione
6/opsx:archive — archivia in archive/YYYY-MM-DD-nome/
Planning e design generati insieme nella proposta. File separati ma nessun gate. Enfasi sull'iterazione: proponi → implementa → scopri vincolo → aggiorna spec → continua.

6 Confronto del Planning Agent

Planning come Processo Formale

Chi controlla il piano: TU. L'agente genera, tu validi e raffini.

Granularità: Alta. File separati: plan.md, data-model.md, api-spec.json, research.md.

Punto di forza: /speckit.clarify forza domande PRIMA del plan — singolo elemento che riduce di più il rework.

Punto debole: La cerimonia completa (7 step) è pesante per feature piccole o bug fix.

Planning come Conversazione Naturale

Chi controlla il piano: L'agente propone, tu validi. L'agente decide quando passare avanti.

Granularità: Molto alta nei task (2-5 min, codice completo, file path) ma meno nella documentazione formale.

Punto di forza: Two-stage review (spec + quality) durante esecuzione garantisce aderenza al piano.

Punto debole: Meno tracciabilità. Il design doc è informale. Per audit enterprise potrebbe non bastare.

Planning come Proposta Fluida

Chi controlla il piano: TU, con meno cerimonia. /opsx:propose genera tutto, poi modifichi liberamente.

Granularità: Media. 4 artefatti generati insieme. Meno file rispetto a Spec Kit, ma sufficienti.

Punto di forza: Velocità. Da idea a piano implementabile in un singolo comando.

Punto debole: Senza phase gate, un team indisciplinato potrebbe saltare la validazione.

7 Possono Coesistere nello Stesso Progetto?

Come funziona l'"auto-trigger" di Superpowers: Non c'è un runtime, un daemon, o un hook. Le skill sono prompt iniettati nel contesto dell'agente (via plugin su Claude Code, via file config su opencode/Copilot). L'"auto-trigger" significa che l'agente, leggendo le istruzioni, dovrebbe controllare le skill prima di ogni task. Ma non c'è un log o un indicatore visibile che confermi l'attivazione. L'unico modo per verificarlo è osservare il comportamento dell'agente.
Problema della priorità nel contesto: Quando più tool sono installati, l'agente ha nel contesto sia istruzioni esplicite (slash commands con prompt dettagliati) che istruzioni implicite (skill da attivare autonomamente). I comandi espliciti tendono a "vincere" sull'attenzione dell'agente. Le skill implicite potrebbero restare silenti.
⚠️ TEORICA

Spec Kit + Superpowers

In teoria operano su livelli diversi: Spec Kit pianifica (slash commands espliciti), Superpowers esegue (skill auto-triggered). A livello di file system nessun conflitto (.specify/ vs .claude-plugin/).

Ma in pratica: durante /speckit.implement l'agente segue le istruzioni esplicite di Spec Kit. Le skill di Superpowers (TDD, code review, subagent) competono per l'attenzione nel contesto senza garanzia di attivazione. Possibile conflitto brainstorming vs clarify, e writing-plans vs /speckit.plan.

Come verificare: In sessione pulita chiedi "quali skill di Superpowers hai attive?". Oppure chiedi di implementare qualcosa senza slash commands — se fa brainstorming socratico e TDD, le skill funzionano.

⚠️ BASSA

Spec Kit + OpenSpec

Competono sullo stesso terreno: produzione di specifiche e piani. Spec Kit produce spec.md + plan.md + tasks.md in .specify/. OpenSpec produce proposal.md + specs/ + design.md + tasks.md in openspec/.

Problema: Due set di artefatti paralleli per la stessa feature. L'agente non sa quale segue. Nessuna sincronizzazione tra i due.

⚠️ TEORICA

Superpowers + OpenSpec

Stessa dinamica di Spec Kit + Superpowers. OpenSpec pianifica, Superpowers dovrebbe eseguire con qualità. Nessun conflitto di directory o branching.

Ma: stessa incertezza sull'attivazione delle skill durante /opsx:apply. La fluidità di OpenSpec potrebbe assorbire meglio eventuali "intrusioni" delle skill.

❌ SCONSIGLIATO

Tutti e tre

Conflitto Spec Kit / OpenSpec sugli artefatti + incertezza Superpowers. Troppe istruzioni contrastanti nel contesto dell'agente.
