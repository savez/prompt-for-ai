1 Identità e Filosofia

Spec Kit

Python/uvGitHub OfficialMIT⭐ 81.4k

github.com/github/spec-kit

Filosofia: Spec-Driven Development — le specifiche non sono documentazione usa-e-getta, diventano artefatti eseguibili che guidano direttamente l'implementazione. Il codice è l'output della spec, non il contrario.

Cosa fa in pratica: Una CLI (specify) che scaffolda il tuo progetto con template, script e comandi per il tuo coding agent. Ti guida attraverso 7 fasi sequenziali: scrivi una constitution (principi fondamentali del progetto), poi specifichi COSA vuoi costruire senza pensare al tech stack, poi chiarisci le ambiguità con domande strutturate, poi crei il piano tecnico, fai un consistency check, generi i task, e infine implementi. Ogni fase produce file Markdown versionati in .specify/ — hai sempre una traccia completa di cosa è stato deciso e perché.

Differenziatore chiave: Il sistema di extensions e presets stackabili permette di adattare l'intero workflow a metodologie diverse (Agile, Kanban, DDD, compliance regolamentare) senza toccare il core. Supporta ufficialmente 25+ coding agent.

Superpowers

Shell/JSJesse Vincent (Prime Radiant)MIT⭐ 137k

github.com/obra/superpowers

Filosofia: Agentic Skills Framework — un insieme di "skill" composabili che il tuo coding agent attiva automaticamente in base a quello che stai facendo. Non devi invocare comandi: l'agente rileva il contesto ("stai per scrivere codice", "c'è un bug", "hai finito un task") e attiva la skill giusta da solo.

Cosa fa in pratica: Installi un plugin (su Claude Code via marketplace, su altri agent via config file) che inietta istruzioni comportamentali nel contesto dell'agente. Da quel momento, quando inizi a parlare di cosa costruire l'agente parte con brainstorming socratico. Quando approvi il design, crea un worktree git isolato. Quando dici "go", spawna subagent per ogni task con review automatica a 2 stadi (aderenza alla spec + qualità codice). Durante tutto il processo impone TDD: se scrivi codice prima dei test, lo cancella e ricomincia.

Differenziatore chiave: Subagent-driven development e TDD hardcoded. L'agente può lavorare autonomamente per ore senza deviare dal piano. Nessun altro framework offre questo livello di autonomia. Include anche skill di debugging strutturato a 4 fasi (root cause tracing, defense-in-depth) che nessun altro copre.

Nota importante: Le skill non sono un runtime — sono prompt iniettati nel contesto. L'"auto-trigger" significa che l'agente dovrebbe seguirle, ma non c'è un meccanismo tecnico che lo garantisca. Verificabile solo dal comportamento dell'agente.

OpenSpec

TypeScript/npmFission AIMIT⭐ 37.7k

github.com/Fission-AI/OpenSpec

Filosofia: "Fluid, not rigid. Iterative, not waterfall. Easy, not complex. Built for brownfield, not just greenfield." — Spec-driven ma dichiaratamente anti-cerimoniale. Puoi aggiornare qualsiasi artefatto in qualsiasi momento senza phase gate.

Cosa fa in pratica: Una CLI Node.js (openspec) che dopo openspec init ti dà slash commands per il tuo agent. Il comando centrale è /opsx:propose "nome-feature": in un colpo genera una cartella openspec/changes/nome-feature/ con 4 artefatti (proposal.md, specs/, design.md, tasks.md). Poi implementi con /opsx:apply, verifichi con /opsx:verify, e archivi con /opsx:archive. In qualsiasi momento puoi tornare indietro e modificare qualsiasi file senza ricominciare il workflow.

Differenziatore chiave: Velocità e fluidità. Da "ho un'idea" a "ho un piano implementabile con 4 artefatti" in un singolo comando. Progettato esplicitamente per il brownfield: la struttura per-change isola ogni modifica e permette di scoprire vincoli della codebase durante l'implementazione senza dover rifare la spec. Il profilo expanded aggiunge comandi avanzati come /opsx:ff (salta al prossimo task incompleto), /opsx:onboard (genera overview per nuovi contributor), /opsx:bulk-archive.

B-MAD

Node.js 22+/npmBMad Code LLCMIT⭐ 42.1k

github.com/bmad-code-org/BMAD-METHOD · docs.bmad-method.org

Filosofia: Agentic Agile Driven Development — "Build More Architect Dreams". Invece di un singolo agente che fa tutto, B-MAD simula un intero team agile con agenti AI specializzati: ognuno ha una persona (personalità, principi, stile), comandi propri, e un ruolo preciso. L'idea è che un Analyst agent elicita i requisiti meglio di un agente generico, e un Architect agent progetta meglio se è "solo" un architetto.

Cosa fa in pratica: Installi con npx bmad-method install (interattivo: scegli moduli e agent). Il framework carica 19+ agenti specializzati e 50+ workflow organizzati in un ecosistema modulare: BMM (software development, 8 agenti, 30+ workflow), Game Dev Studio (4 agenti, 20+ workflow), BMad Builder (per creare i tuoi moduli), Creative Intelligence Suite (brainstorming creativo). Il planning è scale-adaptive su 3 track: Quick (solo tech-spec, per bug fix), BMad (PRD + Architecture + UX, per feature), Enterprise (tutto + Security/DevOps/Test strategy). Lo skill bmad-help ti guida in qualsiasi momento.

Differenziatore chiave: Multi-agent con ruoli separati e scale-adaptive intelligence. È l'unico che adatta automaticamente la profondità del planning alla complessità del task. È anche l'unico che va oltre il software: game development, creative writing, business strategy, wellness. L'adversarial review (verifica deliberatamente critica cross-agente) è unica nel panorama. La curva di apprendimento è la più ripida dei quattro, ma è anche il framework più ricco e completo.

3 In Cosa Eccellono

Spec Kit

  • Rigore del processo — 7 step con gate espliciti
  • Clarification workflow — /speckit.clarify rimuove ambiguità prima del plan
  • Extensions/Presets — overrides > presets > extensions > core
  • Walkthroughs enterprise — ASP.NET 307k LOC, Java 420k LOC
  • 25+ agent supportati nativamente

Superpowers

  • Autonomia massima — lavora ore senza deviare
  • TDD hardcoded — cancella codice scritto prima dei test
  • Subagent-driven — two-stage review (spec + quality)
  • Debugging strutturato — 4 fasi, root cause tracing
  • Zero ceremony — nessun comando da ricordare

OpenSpec

  • Fluidità massima — modifica qualsiasi artefatto in qualsiasi momento
  • Brownfield-first — ogni change isolata in cartella
  • Setup istantaneo — npm + init
  • Dashboard visuale — stato delle change
  • Archiviazione — storia ordinata automaticamente

B-MAD

  • Multi-agent reale — ogni agente ha persona, principi, e comandi propri
  • Scale-adaptive — 3 track (Quick/BMad/Enterprise) in base alla complessità
  • Adversarial review — verifica cross-agente deliberatamente critica
  • Ecosystem modulare — Game Dev, Creative Intelligence, Builder per creare i tuoi moduli
  • Non solo software — business strategy, entertainment, wellness

6/speckit.analyze — consistency check
7/speckit.tasks → /speckit.implement
Separazione netta "cosa" vs "come". Phase gate rigidi. Ogni fase produce artefatti versionati.

3Approvi design AUTO: git-worktrees
4Piano auto AUTO: writing-plans
5Dici "go" AUTO: subagent-dev
6Codice AUTO: TDD RED→GREEN→REFACTOR
7Fine AUTO: finish-branch
Zero comandi. L'agente decide le transizioni. Skill sono prompt nel contesto, non runtime.

6 Confronto del Planning

Processo Formale

Controllo: TU. L'agente genera, tu validi.

Forza: /speckit.clarify forza domande PRIMA del plan.

Limite: Cerimonia completa pesante per feature piccole.

Conversazione Naturale

Controllo: L'agente propone, tu validi. Decide quando passare avanti.

Forza: Two-stage review garantisce aderenza al piano.

Limite: Tracciabilità informale. Non basta per audit.

Proposta Fluida

Controllo: TU, con meno cerimonia. /opsx:propose genera tutto.

Forza: Velocità. Idea → piano in un comando.

Limite: Senza gate, rischio di spec incomplete.

Team Agile Simulato

Controllo: Ogni agente ha un ruolo. L'Analyst elicita, il PM formalizza, l'Architect progetta.

Forza: Scale-adaptive: Quick track salta la cerimonia, Enterprise track la espande.

Limite: Curva di apprendimento. 19+ agenti e 50+ workflow da capire. Overhead per task semplici se non usi Quick track.

7 Possono Coesistere?

Come funziona l'"auto-trigger" di Superpowers: Non c'è un runtime o daemon. Sono prompt iniettati nel contesto dell'agente. L'attivazione non è osservabile — verificala dal comportamento (fa brainstorming? impone TDD?).
Problema della priorità: Con più tool installati, i comandi espliciti (slash) tendono a vincere sull'attenzione dell'agente rispetto alle istruzioni implicite (skill). Il comportamento non è deterministico.
⚠️ TEORICA

Spec Kit + Superpowers

Nessun conflitto file system. Ma le skill di Superpowers non hanno garanzia di attivarsi durante i comandi di Spec Kit. Possibile conflitto brainstorming/clarify e writing-plans/speckit.plan.

⚠️ BASSA

Spec Kit + OpenSpec

Competono sullo stesso terreno: due set di artefatti paralleli per la stessa feature. L'agente non sa quale seguire.

⚠️ TEORICA

Superpowers + OpenSpec

Stessa dinamica di Spec Kit + Superpowers. Nessun conflitto directory. Stessa incertezza sull'attivazione skill durante /opsx:apply.

⚠️ SOVRAPPOSIZIONE

Spec Kit + B-MAD

Entrambi producono spec formali (spec.md vs PRD) e piani strutturati. B-MAD ha agenti con ruoli definiti che sovrappongono le fasi di Spec Kit. Due "metodologie complete" nello stesso progetto = confusione.

⚠️ TEORICA

Superpowers + B-MAD

B-MAD ha i propri agenti con persona e principi; Superpowers ha skill auto-triggered. Entrambi vogliono guidare il comportamento dell'agente in modo diverso. Conflitto di "chi comanda" nel contesto. B-MAD ha già il suo QA agent vs il TDD di Superpowers.

✅ POSSIBILE

OpenSpec + B-MAD

L'unica combinazione con potenziale reale: OpenSpec per change leggere e veloci (bug fix, small feature) dove B-MAD sarebbe overkill. B-MAD per feature complesse dove serve il team agile simulato. Usati in alternativa sullo stesso progetto, non contemporaneamente sulla stessa feature.

❌ SCONSIGLIATO

Tutti insieme

Troppe istruzioni contrastanti. Quattro metodologie che competono per l'attenzione dell'agente.

8 Casi d'Uso: Quale Scegliere e Perché

Come leggere questa sezione: Ogni scenario è un momento concreto del tuo lavoro quotidiano. Per ognuno ti dico quale tool usare, perché quello e non gli altri, e cosa perdi se scegli diversamente. In fondo trovi il decision tree riassuntivo.

Scenario 1 — Devi costruire un nuovo servizio da zero (greenfield) e i requisiti sono vaghi

→ Spec Kit

Situazione tipica: Il PM ti dice "serve un servizio che gestisce le prenotazioni". Non sai ancora quante API servono, quale DB, come gestire le notifiche. I requisiti sono incompleti e ambigui.

Perché Spec Kit: La sequenza /speckit.specify (scrivi COSA senza pensare al tech) → /speckit.clarify (l'agente ti fa domande fino a quando ogni ambiguità è risolta) → /speckit.plan (ORA parli di tech stack) è progettata esattamente per questo. La fase di clarify è il suo vero superpotere: ti forza a decidere cose che altrimenti scopriresti a metà implementazione (es: "cosa succede se un utente prenota due volte?" — se non te lo chiedi ora, lo scopri dopo con un bug).

Perché NON gli altri:

• Superpowers — fa brainstorming, ma non produce artefatti formali. Se tra un mese devi capire perché hai scelto Postgres invece di DynamoDB, non hai traccia.

• OpenSpec — genera tutto con un comando, ma non ti forza a chiarire. Se i requisiti sono vaghi, /opsx:propose produce spec vaghe. Garbage in, garbage out.

• B-MAD — funziona bene (l'Analyst agent elicita i requisiti), ma la curva di apprendimento è ripida. Se non hai mai usato B-MAD, imparare 19 agenti mentre cerchi di partire un progetto nuovo rallenta tutto.

Scenario 2 — Devi aggiungere una feature a un progetto esistente (brownfield quotidiano)

→ OpenSpec

Situazione tipica: Hai una codebase che funziona. Devi aggiungere dark mode, un nuovo endpoint API, o integrare un servizio esterno. Conosci già il codice.

Perché OpenSpec: Fai /opsx:propose "add-dark-mode" e in 30 secondi hai proposal, spec, design e task nella cartella openspec/changes/add-dark-mode/. Mentre implementi scopri che il CSS è più incasinato del previsto? Aggiorni il design.md senza ricominciare. Nessun phase gate ti blocca. Finito? /opsx:archive e la change è archiviata pulita.

Perché NON gli altri:

• Spec Kit — per aggiungere dark mode devi fare constitution → specify → clarify → plan → analyze → tasks → implement. 7 step per una feature che sai già come fare. È overkill.

• Superpowers — non produce artefatti di specifica. Se lavori in team e un collega deve capire cosa hai fatto, non c'è un proposal.md da leggere.

• B-MAD — puoi usare il Quick track, ma devi comunque caricare un agente, capire quale track attivare. Per un task che sai già fare, è overhead.

Scenario 3 — Hai 3 ore di focus time e vuoi che Claude Code implementi senza fermarti

→ Superpowers

Situazione tipica: Hai già chiaro cosa costruire (magari hai già una spec scritta a mano). Vuoi che l'agente lavori in autonomia: scriva test, implementi, faccia review, e vada avanti senza chiederti conferma ogni 2 minuti.

Perché Superpowers: L'agente spawna un subagent per ogni task, fa review a 2 stadi (aderenza alla spec + qualità codice), e se qualcosa non va lo blocca da solo senza aspettarti. Il TDD hardcoded garantisce che ogni pezzo di codice abbia il suo test. Puoi letteralmente andare a fare altro.

Perché NON gli altri:

• Spec Kit — /speckit.implement esegue i task in sequenza ma non ha subagent, non ha review automatica tra task, non impone TDD. È un esecutore, non un agente autonomo.

• OpenSpec — /opsx:apply è simile: esegue task ma senza intelligenza di revisione autonoma.

• B-MAD — ha multi-agent con handoff, ma richiede checkpoint umano tra le fasi. Non è pensato per "lascio lavorare l'agente 3 ore".

Attenzione: Funziona al meglio SOLO su Claude Code (plugin nativo). Su opencode/Copilot le skill sono prompt nel contesto e non c'è garanzia che si attivino. Verifica dal comportamento.

Scenario 4 — Progetto complesso con molti stakeholder e requisiti articolati

→ B-MAD (BMad o Enterprise track)

Situazione tipica: Stai costruendo una piattaforma con frontend, backend, API, database, e ci sono requisiti di sicurezza, performance, UX. Più persone devono capire le decisioni architetturali. Servono PRD, user stories, architecture doc.

Perché B-MAD: Ogni ruolo del tuo team reale ha un agente AI corrispondente. L'Analyst elicita i requisiti con tecniche avanzate. Il PM genera il PRD con user stories. L'Architect progetta l'architettura. Il Dev implementa story per story. Il QA valida con adversarial review. Ogni agente produce artefatti che le persone reali (product owner, tech lead) possono leggere e validare.

Il vero differenziatore è il scale-adaptive: se hai solo bisogno di un tech-spec → Quick track. Se serve il pacchetto completo → BMad track. Se ci sono vincoli enterprise (security, compliance, DevOps) → Enterprise track. Nessun altro framework ti dà questa granularità di scelta.

Perché NON gli altri:

• Spec Kit — produce spec e plan buoni, ma con un singolo agente che fa tutto. Non c'è separazione dei ruoli. L'Architect e il PM sono lo stesso agente → rischio di compromessi non espliciti.

• Superpowers — eccelle nell'esecuzione ma non nella pianificazione strutturata. Nessun PRD, nessun architecture doc formale.

• OpenSpec — troppo leggero per questo livello di complessità. Un /opsx:propose non produce un architecture doc che un tech lead può revieware.

Scenario 5 — Bug fix o task piccolo e chiaro

→ B-MAD (Quick track) oppure nessun framework

Situazione tipica: "Il bottone di logout non funziona su mobile" o "aggiungi un campo email alla form di contatto". Sai già cosa fare, serve solo un minimo di struttura per non dimenticare l'edge case.

Perché B-MAD Quick: È l'unico che ha un track esplicito per task piccoli. Genera un tech-spec minimale (niente PRD, niente architettura) e passa al codice. È la "cerimonia giusta" per il task giusto.

Perché nessun framework: Se il bug è "typo nel testo del bottone", qualsiasi framework è overhead. Fixa, testa, committa.

Perché NON gli altri:

• Spec Kit — 7 step per un bug fix? No.

• Superpowers — potrebbe partire con brainstorming socratico su un typo. Inutile.

• OpenSpec — /opsx:propose è veloce ma genera comunque 4 file (proposal, spec, design, task) per un bug fix di 5 righe.

Scenario 6 — Hai un bug complesso in produzione e non capisci la root cause

→ Superpowers

Situazione tipica: Un endpoint ritorna 500 in modo intermittente. Hai guardato i log, hai provato a riprodurre, ma non riesci a isolare la causa.

Perché Superpowers: È l'unico con una skill di debugging strutturato a 4 fasi: (1) raccolta evidenze, (2) formulazione ipotesi, (3) verifica sistematica, (4) fix con defense-in-depth. Include root cause tracing e condition-based waiting (aspetta che la condizione si verifichi invece di fare sleep random). Gli altri framework non hanno nulla di equivalente.

Perché NON gli altri: Spec Kit, OpenSpec e B-MAD non coprono il debugging come workflow. Al massimo dici all'agente "fixa questo bug" e speri che funzioni. Superpowers ha un processo.

Scenario 7 — Devi dimostrare compliance o tracciabilità delle decisioni

→ Spec Kit con presets compliance

Situazione tipica: Audit, certificazioni, regolatorio. Qualcuno ti chiede "perché avete scelto questa architettura?" e devi poter rispondere con un documento.

Perché Spec Kit: Constitution (principi fondamentali) → Spec (requisiti) → Plan (decisioni tecniche) → Tasks (cosa è stato implementato) — tutto versionato in Git. I presets possono aggiungere gate regolatori (V-Model, traceability). La checklist /speckit.checklist valida i requisiti come "unit test per l'inglese".

B-MAD come alternativa: L'Enterprise track con adversarial review è valido, ma Spec Kit ha un ecosistema di presets compliance più maturo e un'organizzazione (GitHub) dietro.

Scenario 8 — Lavori da solo su side project e vuoi struttura minima

→ OpenSpec

Situazione tipica: Side project personale, nessun team, nessun compliance. Vuoi evitare il caos del "vibe coding" ma non vuoi passare un'ora a configurare framework.

Perché OpenSpec: npm install -g @fission-ai/openspec && openspec init e sei operativo. 3 comandi da sapere (propose, apply, archive). Quando il side project cresce, hai già gli artefatti organizzati per feature.

Perché NON gli altri:

• Spec Kit — richiede Python/uv, constitution, 7 step. Per un side project sabato pomeriggio è troppo.

• Superpowers — buono per l'esecuzione ma non ti dà artefatti organizzati da riprendere lunedì.

• B-MAD — 19 agenti per un to-do app? No.

Scenario 9 — Progetto non-software (game dev, brainstorming creativo, business strategy)

→ B-MAD (unica opzione)

Perché: È l'unico dei quattro che va oltre il software. Game Dev Studio (4 agenti, 20+ workflow), Creative Intelligence Suite, BMad Builder. Gli altri tre sono esclusivamente software development.

🌳 Decision Tree Rapido

Il task è un bug fix di 5 minuti? → Nessun framework, oppure B-MAD Quick

Stai aggiungendo una feature a codice esistente?

Feature chiara e piccola/media → OpenSpec

Feature complessa che cambia l'architettura → B-MAD (BMad track) oppure Spec Kit

Stai partendo un progetto nuovo da zero?

Requisiti vaghi, devi chiarirli → Spec Kit (la fase clarify è decisiva)

Progetto complesso con molti ruoli → B-MAD (Enterprise track)

Prototipo rapido → OpenSpec o nessun framework

Vuoi che l'agente lavori in autonomia per ore? → Superpowers (solo Claude Code)

Devi fare debugging di un issue complesso? → Superpowers

Servono audit/compliance/tracciabilità? → Spec Kit con presets

Non è software? → B-MAD

Side project personale, vuoi struttura minima? → OpenSpec
